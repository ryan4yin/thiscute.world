2024-12-02

  • 学到些经济学新名词:
    • 代际分配: 指上一代人逐渐老去, 他们所占有的资源以及社会上专门为他们服务的资源逐渐被转移到下一代人手中的过程. 它能促进社会资源的合理配置. - 譬如说老一代人喜欢油车, 喜欢自己泡茶, 愿意为了家庭掏空自己的 6 个钱包. 而年轻人更喜欢电车, 喜欢喝奶茶水果茶, 更愿意为了自己的生活质量而花钱. 那么随着代际更迭, 社会资源就会逐渐从油车转移到电车, 从茶叶转移到奶茶水果茶, 从家庭转移到个人.
    • 增量分配: 指社会资源的增加, 以及这些增加的资源如何分配的过程.
    • 古今天下大事, 合久必分, 分久必合. 经济也存在这样的规律.
    • 穷人跟富人的阶级都很稳定, 被夹在他们之间的中产阶级才是最不稳定、最没有安全感的群体, 他们也最容易被经济波动影响.
      • 简单的说, 不管经济好还是差, 城里农民工都是吃 10 块钱的猪脚饭或者快餐, 富人都是米其林三星餐厅, 只有中产阶级在经济好的时候有能力升级消费, 经济差的时候被迫降级消费.
  • 以及一些现象
    • 所谓的新自由主义遭反噬, 发达国家推进逆全球化, 美国川普上太, 中国企业出海难度升级.

  • 继续做 TLPI 的习题
  • 读了 TLPI 的上册后, 对很多以前工作中遇到的问题都有了更深的理解. 以前遇到这些问题只会 Google 解决方案, 但实际并不了解问题的本质. 现在真正做到了知其然知其所以然. 举例来说:
    • 要在 shell 中运行一个后台任务, 以前只知道能用 nohup, setsid 等命令, 但并不知道为什么要这么做, 以及这些命令具体是什么功能. 不知道退出 terminal 的时候到底发生了啥导致普通的后台进程被终止.
    • nohup 得用 & 结尾，而 setsid 又不能用 & 结尾. 以前在 setsid 后也加 & 发现完全不 work, 也不知道为啥, 总之后就再也没用过 setsid…
    • SIGHUP 信号，以前只知道能用来 reload nginx/prometheus, 但并不知道信号具体是个啥东西, 还有啥用.
    • 僵尸进程, 还有 docker 的 tini 程序. 以前使用容器时遇到过程超多僵尸进程造成机器响应变得非常慢, 但 CPU/MEM 等各项监控都很正常, 花了很久才搞明白可以在容器里加 tini 或者在 docker run 时加 –init 来解决. 但也并不是很理解为什么使用容器会遇到这个问题, 以及 tini 如何解决了它.
      • 昨晚还专门看了下 tini 的实现, 非常简单, 只有几百行 C 代码, 我甚至觉得自己写一个都没啥难度.
    • 以及 Linux 中线程跟进程两个术语经常被混用, 我一直觉得相当混乱, 但现在也明白了 Linux 中线程实际可以理解为一类特殊的进程, 线程共享了更多的资源从而更轻量.
  • 总之，TLPI 帮我补全了很多知识体系上的缺漏, 让我在未来遇到类似问题时能有机会搞懂问题的本质, 从而更好地解决问题.

      • 当前制作一个3D游戏角色的周期在 30-45 天左右，且需要的步骤较多，参与的人员众多。采用 AIGC 3D 模型可以将周期缩短至半天，且只需要一个人即可完成，这个成本下降是非常可观的。
      • AIGC 模型能够对游戏研发设计过程中成本和时间进行极大的优化，据不完全统计，在游戏的研发成本中美术成本一般占 50% 到 80%。如果一个游戏公司的研发成本占 70%，那么其中的 40% 基本上都可以大幅降低。
      • 如果 AIGC 3D 的发展使游戏的研发成本大大降低造成大量人员与资本涌入，但游戏的需求量不变，那这个行业不是会变得更卷？这可能得从 AR/VR、云社交等方面寻找答案了，如果类似 DLSS、UE5 全局光照之类的技术能应用到 AR/VR 上提高视觉效果，再结合 AIGC 3D 生成的大量高质量 3D 资产的话，或许能帮助人类踏入元宇宙的门槛。
    • 悲观观点
      • AI 生成的 3D 模型几乎不可编辑，修改一个 AI 模型的细节，需要的时间可能比重新做一个模型还要多，这是最大的问题。
      • 大公司主攻的 Sora/可灵等视频大模型技术，从生成的内容上看，它明显具有空间感，或许视频大模型才是 AIGC 3D 的未来发展方向。
      • 数据是 AI 的基础，但 3D 建模领域的公开数据太少了，大部分都是付费资产或者企业内部资产。相对的符合真实物理规律的视频数据却有很多，这也使得视频大模型技术更有优势。

  • 持续迭代使用 NixOS + KubeVirt + FLux2 取代 Proxmox VE 的方案，方案基本调通了，时间主要用在了解决这些问题上：
    • K8s 集群中部署的 Controller/Operator 删除非常麻烦，它们会创建各种 CRs 并添加 finalizer，一旦删除顺序不对就会导致 CR 无法删除，这个问题在 Pulumi 时就遇到过，当时以为 Flux2 可能自动能解决这个。但现在看是 K8s 自身的问题，必须通过手动在 Pulumi/Flux2 等工具中声明好依赖关系，才能保证删除顺序正确。另外 victoria-metircs 等个别 operator，即使删除顺序是正确的，也会因为它设置的许多保护策略导致无法删除，必须手动一个个确认并删除 （繁琐，但我能理解这么做的目的，确保不丢数据）。

    • RFC062 - content-addressed-paths
      • 受到许多 Nix 用户关注的一个实验特性。它从包的 output 计算其 Output path 与对应的 Hash 值，固定的 Output 总是输出到固定的地址。
      • 这一是能减少许多重复的构建（许多包可能因其 Input 不同而被重复构建，浪费资源），二是可以替代掉旧的基于 public key 的缓存信任模型（拉下来的缓存只要地址跟源码能匹配，那就肯定没有被篡改过）。Content-addressable storage 是一个在存储领域被广泛用于数据可信校验与数据去重，比如你在 QQ 里上传一份，有时候发现它几个 G 的数据一两秒就上传成功了，这实际根本没上传数据，只是对比了下文件 Hash 发现 QQ 线上已经有这个文件了。
      • 该特性最大的改变是，以前 Nix 是从 Derivation 的 inputs 计算其 output 路径，包定义也只锁定了 src 源数据 的 Hash 值。因此在执行构建之前，一个 derivation 的 output 路径就已经被确定了。而 CAS 这个特性则必须通过构建结果来计算其 output 路径，因此 output path 必须在构建完毕后才能给出。
      • 这其中可能存在的问题是，有些包的构建结果并不是确定性的，比如你构建两次 glibc，可能会因为时间或其他随机因素而产生不同的结果。

    • RFC146 - 使用类似 tag 的方式对 packages 做分类: Nixpkgs 确实一直缺少一个好的分类方式，在基于文件夹的分类方式已经被[RFC 0140] Simple Package Paths 取代的背景下提出这个特性，我觉得是个非常好的想法，希望它尽快落地！
